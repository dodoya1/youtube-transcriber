# CLAUDE.md

始めに、 @README.md を参照すること。

## 基本的なルール

- 正直であること。
- わからないことがあれば、ユーザーに質問すること。
- コミットコマンドは実行しないで下さい。ユーザーがコミットをします。

## Plan Mode 時のルール

- plan モードで計画を立てて、ユーザーがその計画(plan)に承諾した場合は、計画を実施する前に、必ず `plan` フォルダ内に当該計画を記載したドキュメントを作成すること。
- チェックリスト
  - 計画書内には、チェックリスト(`- [ ]`)を作成すること。
  - 計画を進めながら、計画書内のチェックリストにチェックを入れて完了済みにすること。
- ユーザーから明示的に指示がない限りは、計画書を作成した後、すぐにその計画を実行せずに、ユーザーに計画書の内容を確認してもらうようお願いして下さい。
- **変更を説明すること**: 各ステップで高水準の要約を示すこと。

### 計画書 (plan.md) の構成

計画書の節は必ず以下の順序で記載すること。これ以外の詳細 (ハイクオリティプロンプト、アーキテクチャ、各モジュール詳細、手動検証手順、リスクと対処、実装ステップ等) はこの 4 節の後にまとめて記載する。

1. **概要**
2. **コミット戦略** — 後述のフォーマットに従って記載
3. **テスト項目 (ユーザー向け受け入れチェックリスト)** — ユーザーが実装結果を確認するためのチェックリスト (`- [ ]` 形式)
4. **ディレクトリ構成** — プロジェクトのツリー図
5. その他

### コミット戦略の記述フォーマット

レビューしやすさのため、コミット戦略は以下の固定フォーマットで記載する。

```markdown
## コミット戦略

### コミット1: xxx

- コミットメッセージ: xxx
- コミットの概要: xxx
- 変更ファイル: xxx

<コミットの内容についてもう少し詳しく解説>

### コミット2: xxx

- コミットメッセージ: xxx
- コミットの概要: xxx
- 変更ファイル: xxx

<コミットの内容についてもう少し詳しく解説>

### コミットn: xxx

- コミットメッセージ: xxx
- コミットの概要: xxx
- 変更ファイル: xxx

<コミットの内容についてもう少し詳しく解説>
```

- 各コミットは 1 つの意味のあるまとまり (粒度) で分割し、変更をユーザーがレビューしやすいようにすること。
- `xxx` の部分にはその時点での実際のコミット内容を書く。
- 「変更ファイル」にはそのコミットで変更・追加されるファイルを列挙する。
- 各コミットブロックの末尾には、そのコミットが何を達成するか・なぜその分割にしたかを数行で解説する。

## ワークフロー・オーケストレーション

### Plan Mode のデフォルト

- 3つ以上の手順やアーキテクチャ上の判断を伴う、あらゆる非自明なタスクでは、必ず plan モードに入ること。
- 何かが想定外の方向に進んだら、そこで止まり、直ちに再計画すること。そのまま無理に押し進めてはならない。
- plan モードは、構築時だけでなく検証手順にも使うこと。
- 曖昧さを減らすために、最初に詳細な仕様を書くこと。

### サブエージェント戦略

- メインのコンテキスト・ウィンドウを整理された状態に保つため、サブエージェントを積極的に活用すること。
- 調査、探索、並列的な分析は、サブエージェントに切り出すこと。
- 複雑な問題には、サブエージェントを使ってより多くの計算資源を投入すること。
- 各サブエージェントには、1つの方針だけを担当させ、実行を集中させること。

### 完了前の検証

- 動作することを証明しない限り、タスクを完了扱いにしてはならない。
- 必要に応じて、main と自分の変更との差分の振る舞いを比較すること。
- 自分にこう問いかけること。「スタッフエンジニアならこれを承認するだろうか？」
- テストを実行し、ログを確認し、正しさを実証すること。

### エレガンスを求める（バランス重視）

- 非自明な変更では、いったん立ち止まり、「もっとエレガントなやり方はないか？」と自問すること。
- 修正が場当たり的に感じられるなら、「今わかっていることをすべて踏まえて、エレガントな解決策を実装する」こと。
- 単純で明白な修正では、これを省いてよい。過剰設計はしないこと。
- 提示する前に、自分の仕事を自分で厳しく見直すこと。

### 自律的なバグ修正

- バグ報告を受けたら、そのまま修正に入ること。手取り足取りの指示を求めてはならない。
- ログ、エラー、失敗しているテストを特定し、それを解決すること。
- ユーザーにコンテキスト・スイッチを一切要求しないこと。
- 失敗している CI テストは、やり方を指示されなくても修正しに行くこと。

## 中核原則

- **まず単純さを優先すること**: あらゆる変更を、可能な限り単純にする。影響するコードは最小限に抑える。
- **怠けないこと**: 根本原因を突き止める。一時しのぎの修正はしない。シニア開発者の基準で対応する。
- **影響を最小限にすること**: 変更は必要な箇所にのみ加える。不具合を持ち込まないようにする。
